# Техническое задание на написание статей для блога

## 1. Метаданные статьи (Frontmatter)

```yaml
---
title: "Заголовок статьи"
description: "Краткое описание (1-2 предложения, до 160 символов)"
heroImage: "../../../../assets/imgs/2025/11/slug-statii.webp"
pubDate: "2025-11-12"
---
```

### Поля:

- **title** — полный заголовок статьи, раскрывающий тему
- **description** — краткое описание для SEO и превью
- **heroImage** — путь к изображению (формат `.webp`, относительный путь)
- **pubDate** — дата публикации в формате `YYYY-MM-DD`
другие поля ЗАПРЕЩЕНЫ (включая tags и github)

title (Заголовок):
- Оптимально: 50–60 символов.
- Максимум: До 70–80 символов.
- Совет: Начинайте с главного ключевого слова.

description (Описание):
- Оптимально: 150–160 символов.
- Диапазон: 120–160 символов для хорошего отображения на устройствах.
- Максимум: 200-300 символов.
- Совет: Добавьте призыв к действию (CTA) и УТП. 

heroImage: должен ссылаться на webp файл, который назван так же как и файл статьи (совпадают их slug)
pubDate: должен содержать дату публикации в таком же формате (дата должна совпадать с тем, где расположен файл статьи)

Основные правила:
- Уникальность: Каждый Title и Description должны быть уникальны для каждой страницы сайта.
- Ключевые слова: Используйте главные ключевые слова, но избегайте "переспама".
- CTR: Хорошо написанный, привлекательный Description повышает кликабельность (CTR) в поисковой выдаче.

## 2. Структура статьи

### 2.1. Заголовок (H1)

- Один заголовок H1 в начале статьи
- Должен отличаться от `title` из метаданных

### 2.2. Введение

- 2-4 абзаца вводного текста
- Кратко о проблеме, которую решает тема статьи
- Что читатель получит в результате

### 2.3. Основные разделы (H2, H3)

- Логическая иерархия заголовков
- Нумерация разделов не требуется
- Каждый раздел раскрывает одну тему

### 2.4. Заключение

- Итоги статьи (1-2 абзаца)
- Призыв к действию или ссылки на дополнительные материалы

## 3. Стиль и подача

### 3.1. Тон повествования

- **Профессиональный, но доступный** — без излишнего академизма
- **Прямое обращение** — можно использовать «вы», «мы»
- **Без воды** — каждая фраза несёт информацию

### 3.2. Форматирование текста

- **Жирный текст** — для ключевых терминов и важных мыслей
- *Курсив* — для иностранных терминов, названий
- `Моноширинный текст` — для имён переменных, функций, команд

### 3.3. Списки

```markdown
- Обычный маркированный список
- Пункты с кратким пояснением

1. Нумерованный список
2. Для последовательных шагов
```

### 3.4. Цитаты и выделения

```markdown
> Пример: Слово «PIN» для разработчика мобильных ОС — это код разблокировки экрана...
```

- Для примеров, пояснений, важных замечаний

### 3.5. Таблицы

таблиц в статьях быть не должно. все можно выразить через списки или другие способы оформления

## 4. Код и примеры

### 4.1. Блоки кода

```python
from fastapi import FastAPI

app = FastAPI()

@app.get("/")
def read_root():
    return {"Hello": "World"}
```

- Всегда указывать язык для подсветки синтаксиса

### 4.2. Инструкции в терминале

```bash
# Установка пакета
pip install fastapi uvicorn

# Запуск сервера
uvicorn main:app --reload
```
- Комментарии внутри блоков для пояснения команд

### 4.3. Встраивание кода в текст

- Для коротких имён: функция `get_user()`, параметр `item_id`
- Для значений: установите `debug=True`, верните `{"status": "ok"}`

## 5. Размер статей

- **Не менее 5 000 символов** для полноценной статьи
- **Оптимально 8 000 — 15 000** для большинства тем
- **40 000+** только для исчерпывающих руководств (справочники, документация)

## 6. Чек-лист перед публикацией

- [ ] Frontmatter заполнен полностью
- [ ] Заголовок H1 один в начале
- [ ] Иерархия H2 → H3 соблюдена
- [ ] Код оформлен в блоки с указанием языка
- [ ] Жирный текст только для ключевых моментов
- [ ] Списки используются для перечислений
- [ ] Ссылки рабочие и релевантные
- [ ] Орфография и пунктуация проверены
- [ ] Размер статьи соответствует рекомендациям

## 7. Запрещённые приёмы

❌ **Не использовать:**
- Разговорный сленг («костыль», «костылить», «говнокод»)
- Излишнюю эмоциональность («это просто пушка!», «лучшее решение ever»)
- Воду и общие фразы без конкретики
- Длинные вступления без сути
- Копипаст документации без переработки

✅ **Использовать:**
- Профессиональную терминологию
- Конкретные примеры и цифры
- Структурированную подачу
- Практическую пользу для читателя
