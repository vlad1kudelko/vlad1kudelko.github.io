# Почему инфраструктура начинается с репозитория

Git давно вышел за рамки инструмента для хранения исходного кода. В современных командах через репозиторий проходят Dockerfile, Kubernetes-манифесты, Terraform-конфигурации, Ansible-роли, CI/CD-пайплайны, документация и инструкции по эксплуатации. Для DevOps-инженера это такой же базовый рабочий инструмент, как терминал, SSH и система мониторинга.

Главная ценность Git состоит в том, что он фиксирует изменение как проверяемое событие: кто изменил файл, когда это произошло, что именно попало в историю и к какой задаче относилась правка. Когда инфраструктура описана текстовыми файлами, такой журнал изменений становится основой воспроизводимости и контроля.

## Что решает система контроля версий

Без системы контроля версий проект быстро превращается в набор файлов с ручными копиями, архивами и непонятным состоянием. Один инженер меняет конфигурацию Nginx, второй обновляет манифест Kubernetes, третий правит скрипт деплоя. Если эти действия выполняются напрямую на сервере или пересылаются отдельными файлами, команда теряет ответ на главный вопрос: какое состояние считается правильным.

Git сохраняет историю изменений и позволяет вернуться к любой зафиксированной версии. Каждый коммит становится снимком проекта в конкретный момент времени. Это удобно для кода, но еще важнее для инфраструктуры: ошибочную правку можно найти через историю, откатить отдельным коммитом или восстановить рабочую конфигурацию после сбоя.

Репозиторий в этом контексте — каталог проекта вместе со всей историей. Внутри могут лежать исходники приложения, Helm-чарты, Terraform-модули, Docker Compose-файлы, playbook для Ansible и runbook для дежурного инженера. Git отслеживает их как обычные текстовые файлы, поэтому инфраструктурные изменения получают те же механизмы проверки, что и код приложения.

## GitHub добавляет совместную работу и автоматизацию

Git работает локально и может использоваться без внешнего сервиса. GitHub, GitLab, Bitbucket и Azure DevOps решают другую задачу: дают общее место для хранения удаленного репозитория, совместной проверки изменений и автоматизации процессов.

Для команды удаленный репозиторий становится точкой обмена. Каждый участник работает со своей копией, отправляет изменения через push и получает обновления через pull. Такой подход особенно полезен, когда над инфраструктурой одновременно работают разработчики, DevOps, SRE и специалисты по безопасности.

GitHub также связывает код с процессом разработки. Issues описывают задачи, ошибки и улучшения. Projects помогают вести доску работ. Milestones группируют задачи под релиз или этап. Releases фиксируют готовую версию с понятным списком изменений. Pull Request превращает изменение в проверяемое предложение: команда видит diff, обсуждает архитектурные решения, запускает тесты и принимает правку осознанно.

Для DevOps это означает, что настройка мониторинга, обновление Terraform-провайдера, добавление резервного копирования или изменение CI/CD-конвейера проходят через тот же прозрачный маршрут, что и продуктовая фича.

## Базовый цикл работы

Практический цикл обычно выглядит просто: получить актуальную версию, внести изменение, проверить результат, зафиксировать коммит, отправить его в удаленный репозиторий. В командах поверх этого добавляются ветки и Pull Request.

```bash
git pull
git status
git add .
git commit -m "Добавить резервное копирование"
git push
```

`git pull` обновляет локальную копию. `git status` показывает измененные, новые и подготовленные файлы. `git add` помещает нужные изменения в staging area. `git commit` создает запись в истории. `git push` отправляет локальные коммиты в удаленный репозиторий.

Ветки позволяют изолировать работу над задачей. Инженер может менять конфигурацию CI, обновлять Kubernetes-манифесты или готовить миграцию Terraform отдельно от основной ветки. После проверки ветка объединяется с основной линией разработки через Pull Request. Если эксперимент оказался неудачным, его можно закрыть без влияния на стабильную конфигурацию.

## Infrastructure as Code требует дисциплины Git

Ручная настройка серверов выглядит приемлемой только на маленьком масштабе. Один сервер можно обновить по SSH, запомнить последовательность команд и записать часть действий в заметки. На десятках серверов такой подход приводит к расхождению конфигураций, ошибкам и невозможности быстро восстановить окружение.

Infrastructure as Code предлагает описывать желаемое состояние инфраструктуры файлами. Вместо устной инструкции появляется декларативная конфигурация: какие ресурсы нужны, какие пакеты установлены, какие порты открыты, какие переменные передаются в сервисы, сколько реплик должно работать в кластере.

Git усиливает IaC тремя свойствами:

- воспроизводимостью: конфигурацию можно применить повторно в новом окружении;
- проверяемостью: изменение видно в diff до применения;
- историей: у каждой правки есть автор, дата, сообщение коммита и связанная задача.

Если конфигурация хранится в репозитории, команда получает понятный источник истины. Сервер, кластер или облачная среда должны соответствовать тому, что описано в Git. Когда фактическое состояние расходится с репозиторием, проблема находится в окружении или в процессе применения изменений.

## GitOps автоматизирует применение изменений

GitOps развивает идею Infrastructure as Code. Репозиторий становится не только местом хранения конфигураций, но и управляющим интерфейсом для инфраструктуры. Инженер меняет файл, создает коммит, отправляет его в репозиторий, а дальше автоматизация проверяет и применяет новое состояние.

Например, изменение количества реплик сервиса в Kubernetes проходит через pull request. CI может проверить YAML, выполнить policy-check, прогнать тесты Helm chart и убедиться, что манифесты корректны. После слияния GitOps-контроллер, например Argo CD или Flux, замечает новое состояние и синхронизирует кластер.

Такой процесс снижает зависимость от ручных действий. Для отката достаточно вернуть предыдущую конфигурацию через Git. Для аудита достаточно посмотреть историю коммитов и обсуждение в Pull Request. Для восстановления окружения можно снова применить файлы из репозитория.

## Как закрепить навык на практике

Лучший способ освоить Git для DevOps — использовать его с первого дня в любом учебном или домашнем проекте. Можно создать небольшой репозиторий с Docker Compose, добавить Issue для настройки HTTPS, оформить ветку с изменениями, открыть Pull Request, связать коммит с задачей и создать Release с краткими заметками.

Затем стоит добавить простую автоматизацию: проверку форматирования YAML, сборку контейнера, тестовый запуск сервиса или валидацию Terraform. Такой мини-проект показывает полный цикл: задача, изменение, проверка, слияние, выпуск и применение.

Git становится особенно полезным, когда команда перестает воспринимать его как хранилище файлов. Репозиторий фиксирует договоренность о том, как должна выглядеть система. Для homelab это помогает быстро восстановить сервисы после экспериментов. Для production-инфраструктуры это дает контролируемый процесс изменений, понятную историю и основу для GitOps-подхода.
